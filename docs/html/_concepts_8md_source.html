21 ## Toolchain
22 Allow the user to define custom toolchains. This allows a project to completely focus on building for either the Web with Emscripten/Cheerp/WASM or even use a custom compiler install like HHVM does!
23 
24 This allows great flexibility.
25 
26 Features:
27 - Inherit the steps of the parent
28 - Mark compatible to other toolchains. For instance, a project compiled with your custom clang compiler may actually still be compatible to the native compiler.
29 
30 ```javascript
31 toolchain("native") {||
32  step(...)
33  step(...)
34  rule(...)
35 }
36 toolchain("emscripten") {||
37  step(...)
38  rule(...)
39 }
40 toolchain("cheerp", "native") {|super|
41  // super is the scope of the parent.
42  step(...)
43  step(...)
44  rule(...)
45 }
46 toolchain("custom-clang") {||
47  // ...
48  this.markCompatible("native");
49 }
50 
51 target("foo", "exe") {
52  toolchain: "cheerp",
53  // ...
54 }
55 
56 target("l1", "lib") {
57  input: [...]
58 }
59 target("l2", "lib") {
60  toolchain: "cheerp",
61  input: [...]
62 }
63 target("bar", "exe") {
64  needs: [
65  "l1",
66  "l2" // <- Error: Target bar can not depend on l2: Toolchains dont match!
67  // Make sure that either they inherit, marked compatible, or are the same!
68  ]
69 }
70 ```

125 ## Project relationships
126 - Refference nested targets with `::`.
127  - In this example, we'd have these targets:
128  * `A::a`
129  * `B::b`
130 ```javascript
131 project({
132  name: "A",
133  version: "1"
134 }) {||
135  target("a", "lib") {
136  // ...
137  }
138 }
139 project({
140  name: "B",
141  version: "1"
142 }) {||
143  target("b", "exe") {
144  needs: ["A::a"]
145  }
146 }
147 ```

150 ```javascript
151 // Much like require, but:
152 project("./deps/minihttp");

170 ## External packages (ITPM, IceTea Package Manager!!)
171 ```javascript
172 target("example", "exe") {
173  packages: {
174  // Installs packages via the package manager.
175  // The package manager to be "emulated" goes before the name
176  // with a colon.
177  // For instance:
178  "clib:jwerle/fs.c": "0.1.1",
179  "conan:Poco": "1.6.1@lasote/stable"
180  },
181  input: pfs.glob("./src", "*.c") + pfs.glob("./src", "*.cxx")
182 }
183 ```
184 
185 In this case, the actual target will change slightly. The packages will be downloaded before the configure step. If they contain a `build.it` file, that one will be run after all the packages are downloaded. Once that's done, we resolve the package names into target names. A target should tag itself with the package name that it was installed as.
186 
187 ```javascript
188 target("fs_c", "lib") {
189  tag: "jwerle/fs.c",
190  input: [...]
191 }
192 ```
193 
194 The tags are then added to the `needs` array.
195 
196 ```javascript
197 // It could become this:
198 var target = IceTea.__targets.example;
199 target.needs = target.needs || [];
200 target.needs += IceTea.tag("jwerle/fs.c");
201 target.needs += IceTea.tag("Poco");
202 ```
203 
204 However, the package is __not__ required to export it's target via tagging. Instead, it can also use the project-target architecture, and have a `package.it` file that could look like this:
205 
206 ```javascript
207 package("jwerle/fs.c") {
208  provides: ["jwerle::fs.c"],
209  dependencies: {
210  // A package can depend on others too.
211  // If this is found, then we re-run the process as often as needed
212  // untill we have all packages.
213  // HOWEVER: Due to limitations in C and C++, we can only have one
214  // version of a package installed!
215  // So if two packages want two incompatible dependencies, that wont work.
216  }
217 }
218 ```
219 
220 In this case, we simply look up the package, see what it provides and add that into the target that wanted it!
221 
222 But even if that is not provided, the user is the poor one that has to specify it himself... :). However, that might also be useful. Imagine you are having a package that provides two versions: One with backwards compatibility and one that breaks the ABI. Take for example FLTK. By defining the `FL_ABI_VERSION` macro, you can basically kill BC. So what you could do:
223 
224 ```javascript
225 target("app", "exe") {
226  packages: {
227  "git:fltk/fltk": "#master"
228  },
229  needs: [
230  "FLTK@abi-1.4::FLTK"
231  ]
232 }
233 ```
234 Boom. Just like that, we will compile with an abi-breaking FLTK!
235 
236 But what we might also could do is to allow a package to specify a callback to determine what it wants to add. Taking the example from above, imagine a setting:
237 ```javascript
238 target("app", "exe") {
239  packages: {
240  "git:fltk/fltk": "#master"
241  },
242  settings: {
243  FLTK: {
244  useFutureABI: true
245  }
246  }
247 }
248 ```
249 ```javascript
250 // FLTK/package.it
251 package("fltk/fltk") {
252  inject: function(target) {
253  if("FLTK" in target.settings) {
254  if(target.settings.FLTK.useFutureABI) {
255  target.needs += [
256  "FLTK@abi-1.4::FLTK"
257  ]
258  }
259  }
260  }
261 }
262 ```
263 
264 And, yes, you can also specify multiple packages in one `package.it` file, if you published it under various names :).

284 ## Implementing a package manager
285 Now that I have talked about how they could look like, this is how I could imagine them being implemented. Once IceTea has built itself, it has mbedTLS and miniHTTP inside, allowing it to call Github APIs and other neat things.
286 
287 So, take for instance you have a little self-hosted package manager...
288 ```javascript
289 var MyPM = extends AbstractPackageManager {
290  __construct: function() {
291  // The first time it is found, its constructed.
292  },
293  install: function(name, version) {
294  // Install the package. Feel free to re-call this method for sub-instals.
295  // For instance, for installing dependencies of the dependency.
296  },
297  info: function(name, version) {
298  return {
299  // A path to a package.it file
300  package_it: "...",
301  // Information deserialized from a JSON
302  json: {/*...*/},
303  // ... to be extended ...
304  }
305  },
306  isEverythingCompatible: function(deps) {
307  // Return true, if it is safe to install all of these.
308  // VERY. IMPORTANT. NOTE.
309  // If you take NPM, it will have nested dependencies, since JavaScript is not so strict.
310  // ObjectScript also won't mind.
311  // Therefore, each package manager should decide individually if nesting is OK or not.
312  // For instance, there will be two package managers:
313  // - One for plain ObjectScript based stuff in icetea_modules
314  // - Native dependencies in deps/.
315  // Native dependencies cant be mixed - but scripted ones, can.
316  }
317 }
318 PackageManager.add {
319  name: "mypm",
320  description: "My awesome package manager.",
321  version: "0.0.1-dev"
322  class: MyPM,
323 }
324 ```

339 ```javascript
340 package("IngwiePhoenix/demo") {
341  provides: ["demo"],
342  scripts: {

344  preInstall: function(){},

349  // After installation

353  // I.e.: icetea --pm deploy
354  // Instead of running your usual building,
355  // switch IceTea to PackageManager mode so it can
356  // run your scripts.
357  // Also, feel free to use the cli object!
358  deploy: function() {
359  // ...
360  }
361  },
362  // Executed straight after the package file was loaded.
363  load: function(){
364  _G.demo = true;
365  }
366 }

369 ## Install stuff!
370 All the cool kids have it, but we currently don't. So! An install step.
371 - Install arbitary files
372 - Install headers, libraries, executables into a prefix
373 
374 ```javascript
375 target("mylib", "shlib") {
376  input: [/* ... */],
377  settings: {
378  native: {
379  reallocateable: true // -fPIC
380  },
381  install: {
382  // Which headers?
383  headers: pfs.glob("src/", "*.h"),
384  // Or drop the headers all into a relative path.
385  headers: {
386  "mylib": pfs.glob("src/","*.h")
387  },
388 
389  // Since this is a dynamic library, it'll be copied into .../lib
390  // automatically. So we can set some stuff for that
391 
392  // Library name for OSX for instance
393  libraryName: "mylib",
394  // OS X: Install name
395  installName: "@rpath/libmylib.dylib",
396  // OS X: Version to be stanced into the file
397  dylibVersion: "0.0.0"
398 
399  // More options might come.
400  // The rule specified must have an install profile.
401  // It is responsible for installing the library, at that point.
402  }
403  }
404 }
405 ```
