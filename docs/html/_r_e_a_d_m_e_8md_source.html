18 ## Features
19 Here is a short rundown of IceTea's features
20 
21 - __Turn-complete scripting language.__ Uses common and extended syntax, allowing vor a very nice amoutn of syntactic shugar while maintaining a consistency across every single line of code.
22 - __Configuring.__ IceTea can configure your project (aka. your good old friend `config.h`). It can also export this configuration as JSON, so you can use it for more than just IceTea.
23 - __Building.__ IceTea is a build tool by itself. It works via a Rule-Task basis, similar to Ninja.
24 - __Integration.__ You can integrate IceTea as a stand-alone script interpreter into your existing build system, adding either configuration or build abilities to it - or ismply execute neat scripts without worrying for an underlying shell.
25 - __Customizable.__ You can use `bootstrap.it` to override the default rules and tell IceTea how you want things to be handled.
26 - __Dynamic, yet static.__ You can use statically listed or danymically generated input lists, allowing you to choose your prefered method.
27 - __Dependencies.__ You can have many targets that can depend on one another.
28 - __Extensible.__ By creating a local `.IceTea` folder and dropping files with the `.it` extension into it, they will become automatically loaded when IceTea starts, allowing any 3rd-party integration to be included. Because these fiels are sorted, you can use numbers to decide in which orders these extensions are loaded!
29 - __Projects.__ Keep your build organized and your files sorted properly. Projects allow you to sort targets into groups and effectively make your build cleaner.
30 - __Multi-threaded.__ IceTea runs on multiple cores to provide as much speed as it possibly can.

32 ## Building
33 IceTea is ultra, ultra tiny. Therefore there is just one command on UNIX based system and 3 on Windows. Or, `build.sh` on linux and `build.bat`on Windows. The reason is that an ASM macro, `.incbin`, is used to include script files directly into the binary. That does not work so well on Windows.
34 
35 Unix:
36  $ g++ -Wno-switch src/*.cpp -o icetea
37 
38 Windows:
39  > cl deps/incbin/incbin.c /Feincbin.exe
40  > incbin src/scripts.rc -o scripts.cpp
41  > cl /EHsc src/*.cpp scripts.cpp /Feicetea
42 
43 ### A note about Emscripten
44 Yes, IceTea can theoretically be built with Emscripten. Unfortunately, it can't build anything yet due to missing `fork()` support. I will have to code a lot of inline-javascript and utilize `child_process.execSync` or other things in order to get execution going. It won't be easy, but if you want to help me port STLPlus' `SubProcess` classes to support Emscripten, then please let me know!
45 
46 The idea currently is to port `stlplus::*_subprocess` to utilize `child_process`. That means, STLPlus would support using `child_process` in both sync and async flavor. However, that means I'd be limited to Node that supports the `.execSync` stuff. Well, no biggie. :)
47 
48 You can, however, build it as a PoC and run ObjectScript scripts. Do it so:
49 
50 ```
51 # These flags are always used for emscripten. Run from inside the IceTea folder.
52 $ export EMCCFLAGS="-s INVOKE_MAIN=0 --pre-js src/icetea-emscripten.js"
53 $ emcc $EMCCFLAGS deps/incbin/incbin.c -o incbin.js
54 $ node incbin.js src/scripts.rc -o scripts.c
55 $ emcc $EMCCFLAGS -Isrc -Wno-switch scripts.c src/*.cpp -DOS_EMSCRIPTEN -IICETEA_INCBIN_FORCE_EXTERNAL -o icetea.js
56 ```
57 
58 Be aware, that running IceTea requires you to set `--stack-size` to a higher value! ObjectScript's execution can cause a really huge stack size. Invoking IceTea just inside it's own source tree will already cause the stack limit to be exceeded. I found `--stack-size=2048` to work. But you gotta try it out yourself. :)

70 ## It's as easy as opening a bottle of ice tea!
71 ```javascript
72 target("myProg", "exe") {
73  input: pfs.glob("src", "*.cpp"),
74  settings: {
75  native: {
76  includeDirs: ["src/"]
77  }
78  }
79 }
80 ```
81 
82 This will compile all files in `src/` into an executable called `myProg`. It will also create an output folder and build all objects with the prefix `myProg-` - and then link them into the top-level.
83 
84 But what if you wish to run a bit of configuration?
85 
86 ```javascript
87 target("myLib", "shlib") {
88  input: pfs.glob("lib", "*.cxx"),
89  configure: function() {
90  detect.header("stdio.h");
91  detect.header("sys/dirent.h");
92  detect.stdc_headers(); // Checks for all standart C headers.
93  detect.set("MY_NAME", "Value"); // @MY_NAME@ -> Value
94  detect.lib("z");
95  if(detect.lib("curl")) {
96  @settings.LINK.libraries[]="curl";
97  }
98  detect.func("popen"); // Check if we have popen() by default.
99 
100  // *.in -> *. Basically, config.h.in becomes config.h
101  detect.transform("lib/config.h.in");
102  // We can use the following call to create an "actual" header.
103  detect.write_header("config.h");
104  // If you are using another tool to post-process your config, you can...
105  detect.write_json("config-h.json");
106  },
107  settings: {
108  native: {
109  includeDirs: ["lib/"],
110  defines: ["HAVE_CONFIG_H"]
111  }
112  }
113 }
114 ```
115 
116 In this case, before any target is built, the `configure()` method is called upon the target. If it fails, the entire process fails.
117 
118 Now we take it a step further. Imagine you wish to have a conditional configuration.
119 
120 ```javascript
121 target("mySpecialLib", "lib") {
122  input: pfs.glob("static_lib", "*.c"),
123  init: function() {
124  // We can add options to the help screen here!
125  cli.insert {
126  longopt: "--curl-path",
127  arg: "dir",
128  desc: "Path to your libCURL source"
129  }
130  // But wait. We can also do...
131  // If enabled, it'll be true. Otherwise, false.
132  config.enable("foo", "Enable functionality foo");
133  // name, description, default
134  config.with("libmeep", "Use libmeep for a functionality", "no");
135  }
136  configure: function() {
137  if(config.enabled("foo")) {
138  // Do something because foo was enabled.
139  }
140  if(config.with("libmeep")) {
141  if(pfs.isDir(config.withValue("libmeep"))) {
142  // Do something, if the value of --XYZ-with-libmeep is a dir.
143  }
144  }
145  }
146 }
147 ```
148 
149 In the above example, we use the `init()` method to prepare the rest of the target that can not statically be written in. Within `configure()` we check if `--with-libmeep` was specified, and get it's "value". If it has one, it returns it. Otherwise it returns false. But since we gave it a default, "no" is returned. `config.with()` only returns wether it was supplied or not.
150 
151 You don't even need to specify a wildcard for your paths. Imagine you have some test cases and want to build them all!
152 
153 ```javascript
154 for(var _,testfile in pfs.glob("test/", "*.cpp")) {
155  var testname = pfs.basename( testfile );
156  target(testname, "exe") {
157  input: [testfile],
158  title: "MyApp Test: ${testname}",
159  // Get a dependency and add it's exports to this target.
160  needs: ["myMainProject"]
161  }
162 }
163 ```
164 
165 You have multiple ways of getting a dependency: Via the full name, or multiple via a tag:
166 
167 ```javascript
168 target("foo-ext-a", "lib") {
169  // ...
170  tag: "foo-ext"
171 }
172 target("foo-ext-b", "lib") {
173  // ...
174  tag: "foo-ext"
175 }
176 target("bar", "exe") {
177  // ...
178  needs: tag("foo-ext") + ["somethingElse"] // -> foo-ext-a, foo-ext-b, somethingElse
179 }
180 ```
181 
182 Use the plus (`+`) operator to combine arrays, since `tag()` returns all targets matching a tag.

186 ## Inspiration
187 Okay... So I was working a lot with script extensions and had to use either `phpize`, `node-gyp` or something else. If I wanted to port a library to multiple languages, I was quickly caught up rewriting and typing redundant information. So I wanted to be able to do something as simple as:
188 
189 ```javascript
190 target("ttvfs", "lib") {...}
191 target("php-ttvfs", "php-ext") {...}
192 target("nodejs-ttvfs", "nodejs-ext") {...}
193 target("TTVFS", "osx-framework") {...}
194 ```
195 
196 This would build `libttvfs` and then use some extra sources or other settings to build either `TTVFS.framework` for Mac OS X, `ttvfs.so` for PHP and `ttvfs.node` for NodeJS.
197 
198 But that was likely impossible. Some people wrote shims over existing build systems - I have seen some for CMake - but I have a reason why I do not like those other build systems:
199 
200 - CMake:
201  * Weird syntax.
202  * Poor documentation.
203  * Its usually bigger than any project you build with it.
204 - GYP
205  * Requires Python
206  * Has zero configuration methods. I used [cDetect](https://github.com/IngwiePhoenix/cDetect) to get around that.
207 - premake
208  * It looks good, but its syntax doesnt seem very readable to me. Although LUA is very powerful.
209 - Scons
210  * Again, Python.
211  * Slow, and hard to learn.
212  * It's not very clear what files are ment for what and how it has to be maintained.
213 - Tup
214  * Per-Folder files
215  * Very compact syntax doe snot give away what is ment for what.
216 
217 Ninja does not go into the above list, simply because its a pure build tool and its only purpose is to read in generated Ninja files and act upon what they read.
218 
219 Therefore I created IceTea - well, still AM creating it. It runs off [ObjectScript](http://github.com/unitpoint/objectscript) for the most part, uses [STLPlus](https://stlplus.sourceforge.net) for its filesystem and subprocess management, a modified variant of [picosha2](https://github.com/okdshin/PicoSHA2) for hashing files in a cache. There is also my own `cli.h` that supports grouped arguments, whose idea was taken from another argument parser called `NBCL`.
220 
221 IceTea is meant to be compact (currently 900kb when built with `-O3`) and simple. I am not directly aiming for speed, but since you can see `TinyThreads++` in there, you probably guessed already that I want to run build jobs on multiple threads to speed up the build time. I think it's called parallel building...
